  ▎ host-published port 9090 만 보장한다. external 네트워크 의존은 후속 Task에서 도입.

  4. 실행 흐름

  1. 운영자가 docker compose -f docker/prometheus/docker-compose.yml up -d
  2. Prometheus 컨테이너 기동
       ├─ /etc/prometheus/prometheus.yml 로드
       └─ file_sd_configs → /etc/prometheus/targets.yml 읽음
  3. 10초마다 targets.yml 변경 감지 → target set 갱신
  4. 60초마다 각 target의 /actuator/prometheus 로 HTTP GET
  5. 메트릭 수집 → /prometheus(TSDB) 저장
  6. 외부에서 http://localhost:9090/targets 로 상태 조회
  7. (후속) Argus가 http://localhost:9090/api/v1/query 로 PromQL 호출

  targets.yml 변경 시 흐름

  사용자가 docker/prometheus/targets.yml 수정
        ↓ (bind mount, 즉시)
  컨테이너 안 /etc/prometheus/targets.yml 갱신
        ↓ (≤ 10s)
  Prometheus가 파일 재읽기, target set 갱신
        ↓
  새 target에 대해 다음 scrape 주기(≤ 60s)부터 수집

  7. 제외 범위 (이번 Task에서 하지 않음)

  - Argus 측 actuator/prometheus endpoint 노출 (애플리케이션 변경) — 별도 Task
  - Grafana, Alertmanager 설치 / 알림 룰 (alerting:, rules:)
  - Prometheus 보안 설정 (basic auth, TLS, RBAC)
  - 운영용 보존 기간/스토리지 튜닝 (--storage.tsdb.retention.time)
  - Kafka 메트릭 exporter 연동 (Task 002/003 영역과 분리)
  - Argus 컨테이너화 및 동일 docker network 통합
  - file_sd 외 다른 SD 방식 (consul, kubernetes 등)
  - CI 파이프라인에서의 Prometheus 기동
